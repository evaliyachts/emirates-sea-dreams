## Contract and integrity risks

The current interface includes `type`, bathrooms, crew, inclusions, add-ons and tags, but lacks the explicit verified fields required for reliable pages. The same nine “standard inclusions” and four priced add-ons are assigned to all 24 records. These must not be treated as verified per-yacht facts.

Future strict validation should require stable `id`, slug, name, length, guest capacity, year built, hourly AED price, minimum duration and typed media; allow bedrooms, availability, featured and priority only when verified; and reject unknown top-level/media fields, duplicate IDs/slugs, invalid enums/types and media without rights records.

## Content and link risks

- Descriptions are formulaic and repeatedly assign party/corporate/event suitability.
- There is no verified related-yacht algorithm or yacht-to-event contract.
- All pages are orphaned from direct server crawling even though homepage fallback and client UI link to them.
- Titles repeat length/rental wording and need a concise unique template only after public names are approved.

## Required evidence

For each record: official source/owner, fact approver, approval date, price effective date, image rights, exact public name, manufacturer/model spelling, year, capacity, minimum duration, permitted bedrooms field, and any optional availability state. Unsupported bathrooms, crew, inclusions, add-ons and suitability claims should be removed rather than guessed.
